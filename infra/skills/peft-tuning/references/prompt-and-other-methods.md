## Picking a method

These methods split into two families:

| Family | Methods | What is trained | Best for |
|---|---|---|---|
| **Soft prompts** | prompt tuning, prefix tuning, P-tuning, multitask prompt tuning | extra *virtual token* embeddings prepended to the input; base model frozen | very low parameter budget; one frozen backbone serving many tasks |
| **Reparameterization / rescaling** | IA3, OFT, BOFT | small learned vectors (IA3) or orthogonal transforms (OFT/BOFT) applied to existing weights | when you want adapter quality closer to LoRA but with different trade-offs (IA3 = fewer params than LoRA; OFT/BOFT = preserve pretrained knowledge / good for generative image models) |

Rules of thumb:
- Fewest parameters, fastest to train → **prompt tuning**.
- Soft prompts that touch every layer (more capacity) → **prefix tuning**.
- Reusing prompts learned on source tasks for a new task → **multitask prompt tuning**.
- LoRA-like quality with even fewer params, no merge into weights → **IA3**.
- Preserving the base model's behavior (e.g. subject-driven image generation, ControlNet/DreamBooth) → **OFT / BOFT**.

## Shared workflow (all methods)

Every method follows the same three steps: build a config, wrap the model, then train it like any 🤗 Transformers model. Only the config class changes.

```python
from transformers import AutoModelForCausalLM
from peft import get_peft_model

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-3B-Instruct")

# peft_config = <one of the configs below>
model = get_peft_model(model, peft_config)
model.print_trainable_parameters()
# -> trainable params: ... || all params: ... || trainable%: ...
```

`print_trainable_parameters()` is the fastest sanity check that the adapter wrapped correctly — the trainable % should be small (often <1%). Typical next step: hand `model` to a `transformers.Trainer` or a manual training loop exactly as you would the base model.

## Prompt tuning

Learns `num_virtual_tokens` soft-prompt embeddings prepended to the input; the base model stays frozen. Initializing the prompt from real task text (`PromptTuningInit.TEXT`) usually converges faster than random init.

```python
from peft import PromptTuningConfig, PromptTuningInit, TaskType

peft_config = PromptTuningConfig(
    task_type=TaskType.CAUSAL_LM,
    prompt_tuning_init=PromptTuningInit.TEXT,
    num_virtual_tokens=8,
    prompt_tuning_init_text="Classify if the tweet is a complaint or not:",
    tokenizer_name_or_path="bigscience/bloomz-560m",
)
```

- `num_virtual_tokens` is the main capacity knob — more tokens = more parameters.
- With `PromptTuningInit.TEXT`, `prompt_tuning_init_text` is tokenized and must fit within `num_virtual_tokens` (set the token count to roughly the text length); `tokenizer_name_or_path` is required so the text can be tokenized.
- Use `PromptTuningInit.RANDOM` to skip the init text.

Typical next step: wrap with `get_peft_model`, train, then save — the saved adapter is only the prompt embeddings (a few KB).

## Prefix tuning

Like prompt tuning, but the learned prefix is prepended to the keys/values of **every** transformer layer, giving more capacity. An optional MLP reparameterization (`prefix_projection`) stabilizes training.

```python
from peft import PrefixTuningConfig, TaskType

peft_config = PrefixTuningConfig(
    task_type=TaskType.SEQ_2_SEQ_LM,
    num_virtual_tokens=20,
)
```

- `num_virtual_tokens` — length of the prefix.
- `prefix_projection=True` adds a learned MLP that produces the prefix from a smaller embedding (more stable, more trainable params during training).
- `encoder_hidden_size` sets the hidden size of that projection MLP when `prefix_projection=True`.

Typical next step: prefer prefix tuning over plain prompt tuning when prompt tuning underfits.

## P-tuning

Soft prompts generated by a small **prompt encoder** (an MLP or LSTM) rather than learned directly. The encoder can be discarded after training. Configured with `PromptEncoderConfig`.

```python
from peft import PromptEncoderConfig, TaskType

peft_config = PromptEncoderConfig(
    task_type=TaskType.SEQ_2_SEQ_LM,
    num_virtual_tokens=20,
    encoder_hidden_size=128,
)
```

- `encoder_hidden_size` — hidden size of the prompt encoder.
- `encoder_reparameterization_type` chooses the encoder architecture (`"MLP"` or `"LSTM"`).
- `encoder_num_layers` and `encoder_dropout` tune the encoder.

Typical next step: P-tuning was designed for natural-language understanding (classification) tasks; reach for it there.

## Multitask prompt tuning

Trains a single shared prompt across several **source** tasks, then specializes it to a **target** task. Use it when you have multiple related tasks and want transfer between them.

```python
from peft import MultitaskPromptTuningConfig, MultitaskPromptTuningInit, TaskType

peft_config = MultitaskPromptTuningConfig(
    task_type=TaskType.CAUSAL_LM,
    num_tasks=2,
    num_virtual_tokens=50,
    num_transformer_submodules=1,
    prompt_tuning_init=MultitaskPromptTuningInit.TEXT,
    prompt_tuning_init_text="Classify the sentiment of the review:",
)
```

- `num_tasks` — number of source tasks the shared prompt is trained over.
- `prompt_tuning_init` accepts `MultitaskPromptTuningInit` variants such as `TEXT`, `RANDOM`, `AVERAGE_SOURCE_TASKS`, `EXACT_SOURCE_TASK`, and `ONLY_SOURCE_SHARED`.
- To specialize from a trained source prompt, set `prompt_tuning_init` to a source variant and point `prompt_tuning_init_state_dict_path` at the saved source checkpoint.

Typical next step: train on the source tasks first, then re-init from that checkpoint for the target task.

## IA3

(**I**nfused **A**dapter by **I**nhibiting and **A**mplifying **I**nner **A**ctivations.) Learns small vectors that rescale keys, values, and feed-forward activations. Far fewer parameters than LoRA, and — unlike LoRA — IA3 does not merge into a single weight matrix.

```python
from peft import IA3Config, TaskType

peft_config = IA3Config(
    task_type=TaskType.SEQ_2_SEQ_LM,
    target_modules=["k", "v", "wo"],
    feedforward_modules=["wo"],
)
```

- `target_modules` — modules whose activations get a learned rescaling vector.
- `feedforward_modules` — **must be a subset of `target_modules`**; these are treated as feed-forward and rescaled on the output side. The remaining target modules are treated as attention key/value projections.
- `modules_to_save` — extra modules (e.g. a randomly-initialized classification head) to train and save in full.

Typical next step: leave `target_modules`/`feedforward_modules` unset to use the built-in defaults for known architectures; only set them explicitly for a custom model.

## OFT (Orthogonal Fine-Tuning)

Adapts a model by multiplying weight matrices with a learned **orthogonal** transform instead of an additive delta. Because orthogonal transforms preserve angles/norms, OFT retains more of the pretrained model's knowledge — useful for generative/diffusion models.

```python
from peft import OFTConfig, get_peft_model

config = OFTConfig(
    r=8,
    target_modules=["q_proj", "v_proj"],
    module_dropout=0.0,
    init_weights=True,
)
model = get_peft_model(model, config)
```

- `r` — number of OFT blocks (the block-diagonal structure that controls parameter count).
- `module_dropout` — probability of dropping an OFT module during training, for regularization.
- `coft=True` enables **constrained OFT**, with `eps` bounding how far the transform may deviate from identity.
- `block_share=True` shares the orthogonal parameters across blocks to save memory.
- `init_weights=True` initializes the transform to identity so training starts from the unmodified base model.

Typical next step: use OFT when an additive method (LoRA) degrades the base model's behavior too much.

## BOFT (Butterfly OFT)

A more parameter-efficient OFT variant that factorizes the orthogonal transform with **butterfly** matrices. Widely used for DreamBooth / ControlNet adaptation of diffusion models.

```python
from peft import BOFTConfig, get_peft_model

config = BOFTConfig(
    boft_block_size=4,
    boft_n_butterfly_factor=2,
    target_modules=["query", "value", "key", "output.dense", "mlp.fc1", "mlp.fc2"],
    boft_dropout=0.1,
    bias="boft_only",
    modules_to_save=["classifier"],
)
model = get_peft_model(model, config)
```

- Specify **either** `boft_block_size` **or** `boft_block_num`, not both — they are two ways to describe the same block structure, and the other is inferred from the layer dimension.
- `boft_n_butterfly_factor` — number of butterfly factors; higher = more expressive transform.
- `boft_dropout` — dropout on the BOFT modules.
- `bias` — which biases to train (`"none"`, `"all"`, or `"boft_only"`).
- `modules_to_save` — fully-trained extra modules (e.g. a new `classifier` head).

Typical next step: BOFT is the go-to orthogonal method for image-generation fine-tuning; start from the controlnet/dreambooth example configs.

## Saving, loading, and inference

All methods share the same checkpoint flow. `save_pretrained` writes only the adapter weights, not the base model.

```python
model.save_pretrained("output_dir")
```

Reload by re-instantiating the base model and attaching the adapter:

```python
from transformers import AutoModelForCausalLM
from peft import PeftModel

base = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-3B-Instruct")
model = PeftModel.from_pretrained(base, "output_dir")
```

Or let the `AutoPeftModel` classes load both in one call, reading the base model id from the saved adapter config:

```python
from peft import AutoPeftModelForCausalLM

model = AutoPeftModelForCausalLM.from_pretrained("output_dir")
```

Typical next step: call `model.eval()` and generate as usual — soft-prompt and IA3/OFT/BOFT adapters run inline, no merge step required.

## Gotchas

- **`task_type` must match the head.** Use `TaskType.CAUSAL_LM` for decoder LMs, `TaskType.SEQ_2_SEQ_LM` for encoder-decoders, `TaskType.SEQ_CLS` / `TaskType.TOKEN_CLS` for classification. A mismatch silently wires the wrong wrapper.
- **Soft prompts add tokens to the sequence.** `num_virtual_tokens` consumes context length and the prepended tokens shift attention masks/labels — account for them in any custom collator.
- **IA3 `feedforward_modules ⊆ target_modules`** is enforced; listing a feed-forward module that isn't also a target module errors out.
- **BOFT block size vs. block num are mutually exclusive** — set only one.
- **New heads need `modules_to_save`.** A randomly-initialized classifier won't be trained or saved unless you list it; otherwise it reverts to random weights on reload.
- **`print_trainable_parameters()` is the first thing to check** after `get_peft_model` — a surprisingly large trainable % usually means the config targeted the wrong modules.
