# Image build philosophy

How we build agent runtime images and why. Recovering this answer was non-trivial — we burned 5 build cycles on a kitchen-sink lockfile approach before re-anchoring on the canonical pattern. Read this before touching any agent Dockerfile.

## TL;DR

| Decision | Choice | Why |
|---|---|---|
| Base image | `vllm/vllm-openai:v0.9.2` | Bundles torch 2.7 + transformers 4.53 + CUDA runtime in one tested wheel set; matches our GPU task pool's needs |
| nvcc / CUDA toolkit | **Not in image** | Compile-from-source pip packages (torchsort, faiss-gpu, flash-attn) need it; we avoid those packages so we never need the toolkit |
| Package list | **Hand-curated `requirements.txt` (~12 lines)** | One file per agent, generated by grep'ing the agent's import graph |
| Lockfile generation | Not used | Single atomic pip install with full resolution against the base image is simpler and less fragile |
| Install order | `pip install -r requirements.txt` (one command, full resolution) | Tiny package count → conflicts unlikely; trust pip's resolver |
| `--no-deps` | Only for the agent source itself if pip-installed (we vendor instead) | Curated requirements.txt with deps lets pip catch mismatches loudly |
| Compile-at-install packages | **None** | Banned. If a future task needs e.g. flash-attn, we install a prebuilt wheel from PyPI or ditch the dependency |

## The pattern (canonical)

```dockerfile
FROM <base-with-prebuilt-wheels>

# 1. System tools (signal forwarding, envsubst, etc.) — minimal apt
RUN apt-get update && apt-get install -y --no-install-recommends \
        tini ca-certificates curl jq gettext-base procps \
    && apt-get remove -y python3-blinker || true \
    && rm -rf /var/lib/apt/lists/*

# 2. Single atomic pip install of curated package list
COPY requirements.txt /opt/requirements.txt
RUN pip install --no-cache-dir -r /opt/requirements.txt

# 3. Vendored agent source (NOT pip-installed) so we control the version
COPY upstream /workspace/agent

# 4. Our sidecar + entrypoint
COPY sidecar /workspace/sidecar
COPY run_*.py entrypoint.sh /workspace/

# 5. Build-time regression guard
RUN python -c "<assertion that proves a critical claim>"
```

That's it. Three real ingredients: base + curated `requirements.txt` + vendored source. Nothing else.

## Why this works

### From [MLE-Bench's `environment/Dockerfile`](https://github.com/openai/mle-bench/blob/main/environment/Dockerfile) (the field's reference implementation)

MLE-Bench runs the most agent-evaluations of any public project. Their environment is:
- `FROM ubuntu:20.04` (no CUDA image, no nvidia/cuda layer)
- Conda env with **92 hand-pinned packages** in [`environment/requirements.txt`](https://github.com/openai/mle-bench/blob/main/environment/requirements.txt)
- Notably absent: streamlit, autogluon, blinker, umap-learn, torchsort, faiss-gpu, flash-attn, deepspeed, xformers, apex — every package class that compiles at install time
- `torch==2.2.0` installed via prebuilt wheel from default PyPI (bundles its own CUDA runtime)

Their philosophy distilled: *prebuilt wheels only, curated minimal, no compile-time CUDA*. They cite the same problems we ran into when we tried the opposite.

### From our prior [AIDE Dockerfile](https://github.com/Kkuntal990/AI-Skill-builder/blob/main/infra/agents/aide/Dockerfile) (commit `b5c7120`)

Same philosophy, different base. AIDE Dockerfile comment says verbatim:

> Prior: pytorch/pytorch:2.7.1-cu126 + aideml git pin (with old datasets==2.1.0 / rich==13.7.0 pins) + per-task `pip install -r requirements.txt` at trajectory startup → vllm 0.6.6 in per-task reqs forced torch 2.7.1 → 2.5.1 downgrade, broke torchaudio, broke transformers ABI, AIDE judge crashed.
>
> Now: vllm/vllm-openai:v0.9.2 (vllm owns torch ABI) + ONE atomic pip install -r requirements.lock (280 packages, resolved once in this same base image) + AIDE installed with --no-deps so its outdated pins do not re-pin packages the lockfile already owns.

The AIDE base used a generated 280-package lockfile. The MLEvolve base uses an 11-package `requirements.txt`. Same philosophy at different package-list sizes — the principle is *curate*, not *generate-from-upstream*.

### Aira-dojo + MLAgentBench + RD-Agent confirm the pattern

All independently converged on:
- Prebuilt torch + CUDA base (either `pytorch/pytorch:*` or `nvidia/cuda:* + mamba pytorch-cuda`)
- Curated <150-package pin list
- No pip-time CUDA compilation

References:
- [aira-dojo `superimage/apptainer.def`](https://github.com/facebookresearch/aira-dojo/blob/main/superimage/apptainer.def) — mamba installs torch + cuda-toolkit from conda channels, pip only for pure-Python deps
- [MLAgentBench `Dockerfile`](https://github.com/snap-stanford/MLAgentBench/blob/main/Dockerfile) — `FROM anibali/pytorch:2.0.0-cuda11.8-ubuntu22.04` + minimal extras
- [RD-Agent MLE-Bench Dockerfile](https://github.com/microsoft/RD-Agent/blob/main/rdagent/scenarios/kaggle/docker/mle_bench_docker/Dockerfile) — `FROM pytorch/pytorch:2.4.0-cuda12.4-cudnn9-runtime` + mle-bench layer

## What we explicitly rejected (and why)

### Rejected: kitchen-sink upstream lockfile (the MLEvolve trap)

MLEvolve ships a 665-line `requirements_{base,ml,domain}.txt` that pins every transitive dep. We tried installing it on our base and surfaced 5 distinct failures in 5 build cycles:

| Failure | Root cause |
|---|---|
| `pillow==12.0.0` vs `streamlit==1.40.2` (streamlit caps pillow<12) | Upstream lockfile internally inconsistent |
| `pyarrow<21` (autogluon) vs `pyarrow>=21` (datasets) | Upstream lockfile internally inconsistent |
| `umap==0.1.1` no Py3.12 wheel | Upstream typo'd duplicate of `umap-learn` |
| `Cannot uninstall blinker 1.4 — distutils installed project` | Base image has python3-blinker apt package; pip can't safely uninstall |
| `torchsort` CUDA 12.8 vs PyTorch's 13.0 mismatch | Compile-at-install package; our base lacks the matching nvcc toolkit |

The pattern across all 5: **MLEvolve's lockfile was frozen in their CI env and assumes that env**. Reproducing it elsewhere requires either matching their env exactly (impractical, undocumented) or stripping incompatible packages one-by-one (whack-a-mole; we'd lose the "we ship a reproducible env" benefit). Per the upstream behavior, even MLE-Bench's own AIDE adapter doesn't try to consume MLEvolve-style lockfiles — they keep their own curated env and add agents via thin pip installs.

### Rejected: `FROM mlebench-env`

Considered: start from MLE-Bench's own environment image (Python 3.11 + torch 2.2 + their 92-package curated env), layer MLEvolve on top. **Cons that killed it**:

1. **Python+torch version lock-in**. mlebench-env pins Python 3.11 + torch 2.2.0. Our task pool needs vLLM 0.9 (torch 2.7) and may need flash-attn 2.6+ (torch 2.4+). Forking mlebench-env to bump torch defeats the purpose of inheriting their env.
2. **Submission contract mismatch**. mlebench-env assumes Kaggle-shaped CSV submissions graded by `mlebench grade`. Our tasks include `llama-inference` (stdout float metric), `LLM-Merging` (adapter checkpoint), `debug-trl-grpo` (test-pass rate) — none CSV-shaped.
3. **Mount layout conventions diverge**. mlebench-env uses `/home/data`, `/home/submission`, etc.; our PVC convention is `/results/data/<task>/`. We'd need a translation layer.
4. **Not a published image**. mlebench-env has no Docker Hub tag — we'd build it ourselves anyway, which means we maintain an upstream we don't own.

Adopting MLE-Bench's BUILD PHILOSOPHY without their actual image is the right call. We get the wins (curated minimal, no nvcc, prebuilt wheels) without the lock-in.

### Rejected: `pip install --no-deps -r <upstream-lockfile>`

Considered: install MLEvolve's 665-line lockfile literally without resolving transitives. Fast, but every undeclared transitive becomes a runtime `ImportError`. Hard to debug. We'd burn equivalent cycles fixing-forward at runtime as we did at install time.

## How to add a new agent

1. Vendor the agent's source as a submodule under `infra/agents/<name>/upstream` pinned to a specific SHA.
2. Grep the source for top-level imports — that's the candidate list:
   ```bash
   grep -rhE "^(import|from) [a-zA-Z]" infra/agents/<name>/upstream \
       --include="*.py" | awk '{print $2}' | awk -F. '{print $1}' | \
       sort -u | grep -vE '^(your stdlib filter here)$'
   ```
3. Subtract packages your base image already provides (test by trying `python -c "import X"` in the base).
4. Write `infra/agents/<name>/requirements.txt` with what's left. Add version pins only where you have evidence the package's API changed across versions (don't pin defensively).
5. Write the Dockerfile following the canonical 5-step pattern above.
6. Add a build-time smoke `python -c "..."` that imports the same modules the agent loads at startup. This catches "we forgot to install X" before the cluster does.

## How to handle a build conflict

If pip resolves but a wheel fails to build (CUDA mismatch, missing nvcc, etc.):

1. **Identify the offending package** (last line of the pip error).
2. **Decide if the agent actually needs it**: `grep -rn "import <pkg>" infra/agents/<name>/upstream`. Most kitchen-sink lockfile entries are libraries available to LLM-generated code, not to the agent itself.
3. **If not needed by the agent code**: drop from `requirements.txt`. Document the drop in a comment.
4. **If needed**: find a prebuilt-wheel alternative (`faiss-cpu` instead of `faiss-gpu`; specific older version that ships wheels for Py3.12), or pin a CUDA-matching version.
5. **Don't reach for `--ignore-installed` or `--no-deps`** to make the conflict go away. Either fixes the symptom but leaves a fragile install.

## Glossary

- **Curated requirements.txt**: hand-written list of packages, one per line, version-pinned only where necessary. Resolved by pip at build time against the base image.
- **Kitchen-sink lockfile**: an upstream-shipped `requirements*.txt` with hundreds of pinned transitives. Optimized for the upstream's CI env, not portable. Avoid.
- **Vendored source**: agent source COPYed into the image (often via git submodule), not pip-installed. Lets us control what runs without depending on upstream's setup.py.
- **Build-time regression guard**: a `RUN python -c "assert <condition>"` step that fails the build if a critical invariant breaks (e.g., a forbidden package becomes importable). Catches problems before the cluster pulls the image.
