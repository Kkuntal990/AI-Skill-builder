DIMENSION 1 — scope-honesty (most important; severity "block"):
The failure mode: the skill EXCLUDES a whole TASK GENRE that it could legitimately be one step of. This caused a real production failure — an inference/serving skill listed, under a "**NOT for:**" header: "Supervised fine-tuning or LoRA training — use transformers.Trainer". Fine-tuning is a TASK GENRE; a fine-tune-then-evaluate task still needs fast inference to evaluate the trained model, so excluding it makes the agent scope the skill out of the ENTIRE task. That line MUST be flagged.

The decisive test for any limitation/exclusion you see: is the excluded thing a DIFFERENT END-TO-END TASK the skill could be a sub-step of (training, fine-tuning, data preprocessing, evaluation, deployment, retrieval, ...) → FLAG it (severity "block"); or is it a SPECIFIC TECHNIQUE/METHOD inside the skill's OWN task that it happens not to implement → that is fine, do not flag.

FLAG (severity "block"):
- Any "NOT for:" / "not suitable for:" / "don't use for:" / "this skill is only for:" exclusion, and every item under such a header that names a task type, workload, or problem genre — EVEN IF the line ends with "— use <other tool> instead". The "use Y instead" suffix does NOT redeem excluding a whole task genre.
- Any exclusion of a task genre regardless of phrasing or where it appears (training, fine-tuning, evaluation, serving, preprocessing, etc.).
- Any direct contradiction between a "use when ..." trigger and a "not for ..." exclusion.

OK — do NOT flag:
- A POSITIVE, capability-scoped escape-hatch for a specific METHOD WITHIN the skill's own job that it does not implement, NOT placed under a "NOT for" header — e.g. an inference skill saying "For multi-node tensor-parallel serving beyond one host, use Ray Serve instead" (a missing capability inside serving, not a different task genre).
- Do not abstain on a "NOT for" header that lists task types — that is a clear block. Reserve abstention for genuinely ambiguous within-domain capability limits.

DIMENSION 3 — ungated conditional / resource-heavy action (severity "block"):
The failure mode: a workflow presents a CONDITIONAL or RESOURCE-HEAVY technique as if it were an unconditional default, with the precondition stranded in another section (the Decision Tree) or absent.
